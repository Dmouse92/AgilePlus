## 1. Repository Overview

`cliproxyapi-plusplus` (`github.com/kooshapari/CLIProxyAPI/v7`) is a Go-based LLM
reverse proxy that exposes an OpenAI-compatible HTTP surface to downstream tools while
forwarding requests to many upstream provider backends. The proxy is the canonical
multi-provider translation layer for the Phenotype org.

**Language:** Go (primary), with a thin Rust-free SDK layer.
**CLI framework:** Standard library `flag` package (no cobra/urfave) for `cliproxyctl`.
**Server framework:** `gin-gonic/gin`.
**Build entrypoints:**

| Binary | Path | Purpose |
|---|---|---|
| `cliproxyapi` (server) | `cmd/server/main.go` | The proxy daemon |
| `cliproxyctl` | `cmd/cliproxyctl/main.go` | Operator CLI: setup, login, doctor |
| `boardsync` | `cmd/boardsync/main.go` | GitHub Projects board sync tool |
| `releasebatch` | `cmd/releasebatch/main.go` | Release batching helper |
| `mcpdebug` | `cmd/mcpdebug/main.go` | MCP protocol debugger |
| `protocheck` | `cmd/protocheck/main.go` | Protobuf schema validation |
| `codegen` | `cmd/codegen/main.go` | Code-generator for provider registry |
| `fetch_antigravity_models` | `cmd/fetch_antigravity_models/main.go` | Upstream model list fetcher |

## 3. Architecture Summary (T001)

```
[ Downstream Tool ]
       |
       v
  [ gin HTTP server / cmd/server ]
       |
       v
  [ sdk/api/handlers/{openai,claude,gemini} ]
         (inbound format detection + initial translation)
       |
       v
  [ pkg/llmproxy/util.GetProviderName() ]
         (model-to-provider resolution, Pareto routing)
       |
       v
  [ sdk/cliproxy/auth.Manager / Conductor ]
         (auth selection, rate-limit gating, credential round-robin)
       |
       v
  [ pkg/llmproxy/executor/<provider>_executor.go ]
         (provider-specific protocol, auth injection, retry)
       |
       v
  [ Upstream LLM Provider API ]
```

The `ProviderExecutor` interface (defined in `sdk/cliproxy/auth/conductor.go`) is the
core abstraction contract every executor must satisfy:

```go
type ProviderExecutor interface {
    Identifier()      string
    Execute(...)      (Response, error)
    ExecuteStream(...)(*StreamResult, error)
    Refresh(...)      (*Auth, error)
    CountTokens(...)  (Response, error)
    HttpRequest(...)  (*http.Response, error)
}
```

Executors are registered with the `Manager` via `RegisterExecutor(exec)` and dispatched
through the `conductor.go` scheduling loop.

## 4. Gap Analysis (T001)

### Missing Providers (spec target vs. current)

| Spec Target | Status |
|---|---|
| OpenAI (GPT chat completions, embeddings) | Present via `CodexExecutor` + `OpenAICompatExecutor` |
| Anthropic (messages, streaming) | Present via `ClaudeExecutor` |
| Google (Gemini) | Present via `GeminiExecutor`, `GeminiVertexExecutor`, `GeminiCLIExecutor`, `AIStudioExecutor` |
| Mistral | Present via `OpenAICompatExecutor` (registered in codegen registry) |
| Cohere | **MISSING** — not in generated registry or executor list |
| Groq | Present via `OpenAICompatExecutor` (registered) |
| Ollama | **MISSING** — not in registry or executor list |
| Local (vLLM, LMStudio) | **PARTIAL** — helios-cli has `codex-rs/lmstudio` crate; cliproxyapi-plusplus has no local executor. `nim` in registry covers some self-hosted. |

**Summary of gaps:**
- **Cohere** — no executor and not in the generated registry. Needs both a registry entry (OpenAI-compat, `https://api.cohere.ai/v1`) and validation of any Cohere-specific response shape differences.
- **Ollama** — no executor. Ollama exposes an OpenAI-compat endpoint at `http://localhost:11434/v1`. Needs a registry entry and a local executor wrapping `OpenAICompatExecutor` with custom base-URL resolution from config (no API key, custom URL from user config).
- **Local model general case** — no unified local executor. Ollama/LMStudio/vLLM each differ in auth (none) and URL (user-configured). A `LocalExecutor` using dynamic base URL from config covers all three.

### Architectural Gaps

| Gap | Severity | Notes |
|---|---|---|
| No unified retry middleware | Medium | Retry is per-executor (only Antigravity has exponential backoff). A shared `RetryTransport` wrapping all executors would unify behavior. |
| `cliproxyctl` uses `flag` not a structured CLI framework | Low | `flag` is adequate for the current command count (setup, login, doctor). As subcommands grow (T002 target), switching to `cobra` is advisable. |
| Token counting not uniformly implemented | Medium | `CountTokens` is in the interface but implementation depth varies. Needed for accurate rate-limit enforcement. |
| No explicit agentapi-plusplus integration hooks | Medium | `agentapi-plusplus` has its own `cmd/` and `internal/cli/` but no verified wire-up to cliproxyapi as upstream. This is a T001 gap to resolve before T008/T009. |

## 5. Provider Abstraction Design (T002)

### 5a. Existing Contract (Canonical — do not replace)

The `ProviderExecutor` interface in `sdk/cliproxy/auth/conductor.go` is already
complete and correct. Downstream implementations must satisfy all six methods.

### 5b. Recommended `OpenAICompatExecutor` Extensions

The generic `OpenAICompatExecutor` in `openai_compat_executor.go` should be the
**default implementation** for any provider that speaks the OpenAI chat-completions
schema. Cohere and Ollama both qualify. Extend the codegen registry instead of writing
new executor structs unless the provider has protocol deviations.

Pattern for adding a new OpenAI-compat provider:

1. Add entry to `AllProviders` in `pkg/llmproxy/config/provider_registry_generated.go`
   (or the codegen source, let codegen re-emit).
2. Add any provider-specific env-var aliases to the `EnvVars` field.
3. Set `DefaultModels` if the provider uses non-standard model IDs.
4. Run `go generate ./cmd/codegen/...` to regenerate.
5. No new executor file needed unless authentication is non-bearer or response format
   deviates.

### 5c. Cohere — Recommended Approach

Cohere's `/v1/chat` endpoint is OpenAI-compat. Add to registry:

```go
{
    Name:    "cohere",
    YAMLKey: "cohere",
    GoName:  "Cohere",
    BaseURL: "https://api.cohere.ai/v1",
    EnvVars: []string{"COHERE_API_KEY"},
}
```

Verify that Cohere streaming (`text/event-stream`) is identical to OpenAI SSE format
before shipping. If deviations exist (Cohere uses `event: stream-start` preamble), add
a thin `CohereSanitizingTransport` that strips the preamble.

### 5d. Ollama — Recommended Approach

Ollama requires a user-supplied base URL, no API key, and has no cloud registration.
Recommended: a dedicated `OllamaExecutor` that wraps `OpenAICompatExecutor` with:

- `BaseURL` resolved from `config.OllamaBaseURL` (default `http://localhost:11434/v1`).
- `PrepareRequest` that omits the `Authorization` header entirely.
- Model list fetched from Ollama's `/api/tags` instead of `/v1/models`.

```go
type OllamaExecutor struct {
    cfg *config.Config
    inner *OpenAICompatExecutor
}

func NewOllamaExecutor(cfg *config.Config) *OllamaExecutor {
    baseURL := cfg.OllamaBaseURL
    if baseURL == "" {
        baseURL = "http://localhost:11434/v1"
    }
    return &OllamaExecutor{cfg: cfg, inner: NewOpenAICompatExecutor("ollama", cfg)}
}
```

### 5e. Unified Retry Middleware

Add a `RetryingExecutor` decorator in `pkg/llmproxy/executor/retry_executor.go`:

```go
type RetryingExecutor struct {
    inner   ProviderExecutor
    maxRetries int
    retryOn func(err error) bool
}
```

Wrap during executor registration in `api/server.go`. Default policy: retry 3x with
exponential backoff on HTTP 429, 502, 503, 504. Provider-specific policies can
override `retryOn`.

### 5f. agentapi-plusplus Integration Hook

`agentapi-plusplus/internal/cli/` should expose a `ClipProxy(baseURL, apiKey string)`
helper that constructs an OpenAI-compat client pointed at the cliproxy server. This
keeps the integration boundary at HTTP (no shared Go package import required). Wire-up
in T009.

### 5g. CLI Subcommand Structure (cliproxyctl)

Current `cliproxyctl` surface uses the stdlib `flag` package with manual dispatch.
Recommended refactor to `cobra` to support:

```
cliproxyctl
  setup           # interactive setup wizard
  login <provider># authenticate to a provider
  doctor          # health-check
  models list     # list available models (new)
  models test     # test model connectivity (new)
  provider list   # list registered providers (new)
  provider status # show rate-limit state per provider (new)
  server start    # start proxy daemon (new)
  server stop     # graceful shutdown via management API (new)
```

This command tree makes cliproxyctl the unified control plane for the proxy and
eliminates the need for separate shell scripts in downstream repos.

## 7. Summary Verdict

**cliproxyapi-plusplus substantially meets the WP-001 acceptance criteria today:**

- 17+ provider executors (well above the 8+ target).
- `ProviderExecutor` interface is clean and extensible.
- Rate limiting, streaming, Pareto routing, and auth round-robin are all implemented.
- Two provider gaps: **Cohere** and **Ollama** — both are straightforward additions.
- No unified retry middleware — medium-priority gap for T007/T008.
- `cliproxyctl` CLI needs `cobra` migration before expanding the command surface.
- agentapi-plusplus integration needs explicit HTTP-level wire-up (T009).

**T003–T006 can proceed using `OpenAICompatExecutor` extension pattern for Cohere/Ollama
and validation-only work for the remaining 6 providers already present.**
